# Royal Bellion

Royal Bellion is an arcane terminal for D&D 5e creature management and monster combat. Dungeon Masters can store creature templates, import JSON stat blocks, roll dice with auditable output, and run monster attacks against player-facing armor class targets.

## Features

- **Creature library** — search and filter by name, type, and CR
- **Manual creation** — full stat block form for custom creatures
- **JSON import** — validate, preview, and save Royal Bellion-format stat blocks
- **DB backup import/export** — save and restore creatures and encounters as a Royal Bellion JSON backup
- **Dice sandbox** — test expressions, d20 rolls, advantage, and disadvantage
- **Quick combat** — resolve monster attacks against an optional target AC with resistance modes
- **Encounters** — build a monster party, select attackers and actions, run rounds against a shared target AC
- **Combat telemetry** — highlight the latest roll batch, show critical misses, and total damage hits against target AC
- **Local MongoDB** — Docker Compose runs a Royal Bellion-named MongoDB database for development

Creature IDs are generated by Royal Bellion as UUIDs. Backup import preserves existing IDs so encounter references remain stable.

Encounter IDs are generated as UUIDs. Encounter combat rolls are grouped by batch so the most recent run is highlighted in the battle feed.

### Combat behavior

Target AC is optional in quick combat and encounters.

- When Target AC is on, attacks resolve as hit, miss, critical, or critical miss. The latest batch shows a Total Damage bar that sums damage from hits and criticals only.
- When Target AC is off, Royal Bellion still rolls to-hit and damage for the DM, but it does not show a final total-damage bar because the DM applies the result manually.
- Natural 1 is shown as critical miss. Natural 20 rolls critical damage by doubling dice only, not modifiers.

Multiattack is configured on an action with an `attacks` list that references other actions:

```json
{
  "name": "Multiattack",
  "description": "The owlbear makes two attacks: one with its beak and one with its claws.",
  "attacks": [
    { "actionName": "Beak", "count": 1 },
    { "actionName": "Claws", "count": 1 }
  ]
}
```

Each referenced attack creates its own log entry in the same batch.

## Getting started

### Prerequisites

- Node.js 20+
- pnpm 9+
- Docker (for local MongoDB)

### Setup

```bash
pnpm install
cp .env.example .env
docker compose up -d
pnpm validate:mongo
pnpm validate:fixtures
pnpm dev
```

Open [http://localhost:3000](http://localhost:3000).

### Environment

| Variable | Description |
|----------|-------------|
| `APP_PASSWORD` | Shared password required to access the app. |
| `SESSION_SECRET` | Long random secret used to sign the 30-day session cookie. Rotate it when revoking all active sessions. |
| `MONGODB_URI` | MongoDB connection string. Local default: `mongodb://royal_bellion:royal_bellion@127.0.0.1:27017/royal_bellion?authSource=admin` |
| `MONGODB_DB` | Database name. Local default: `royal_bellion` |

See [`.env.example`](.env.example) for defaults.

Generate a session secret with:

```bash
openssl rand -base64 48
```

Royal Bellion uses one shared app password. Everyone with the password can view, create,
edit, delete, import, and export data. To revoke access, change `APP_PASSWORD` and
`SESSION_SECRET`, then redeploy so every browser must log in again.

### Private deployment

For a small group, deploy the Next.js app to Vercel and use MongoDB Atlas for the database.
Set these environment variables in Vercel before deploying:

```bash
APP_PASSWORD=your-shared-table-password
SESSION_SECRET=your-generated-session-secret
MONGODB_URI=mongodb+srv://...
MONGODB_DB=royal_bellion
```

Only share the Vercel URL with people who should have the shared password.
